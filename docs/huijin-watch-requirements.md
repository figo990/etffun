## 1. 背景与目标

### 1.1 背景

视频《复刻国家队 - 助力汇金》提出了一套基于公开 ETF 数据观察中央汇金 ETF 持仓节奏的框架。该框架的核心不是直接读取汇金账户真实交易，而是用基金定期报告披露的持有人数据作为基准，再用交易所每日 ETF 总份额变化估算一个可能区间。

`etffun` 当前已经具备全市场 ETF 份额、行情、净值、估值、融资、北向资金等数据采集和展示能力，也已有 `huijin_config.json` 与“比汇金改变比”字段。但当前数据结构不足以严谨实现视频公式，原因包括:

- `huijin_config.json` 只保存汇金持股合计和文字来源，缺少披露日总份额 `S0`、披露比例 `A`、公告日期、原始文档链接和校验状态。
- `daily_snapshot.total_shares` 可以作为当前总份额 `S1` 的候选来源，但还缺少源日期、源单位、交易日校验和异常跳变审计。
- 当前 `fund_holding` 表表示 ETF 持有的股票组合，不是 ETF 持有人披露，不能用于汇金基准。
- 现有前端文案“国家队增仓”“汇金动态调仓”容易被理解成对汇金真实买卖的确认，需要降级为份额观察类表达。

### 1.2 产品目标

新增“汇金观察”能力，用于:

- 基于公开披露数据建立汇金 ETF 持仓基准。
- 基于交易所 ETF 份额数据计算视频中的归一化仓位区间。
- 标记数据缺失、源日期异常、周末落库、单位不明、份额断档、异常跳变等质量问题。
- 在前端提供“汇金 ETF 份额观察”和“份额变动强度”视图。
- 为后续回测、同指数 ETF 池观察和辅助信号验证保留结构化数据基础。

### 1.3 产品边界

该功能是 ETF 候选筛选和公开数据观察工具，不构成个性化投资建议，不输出确定买卖指令。

必须明确:

- 不预测中央汇金真实账户。
- 不把 ETF 份额变化直接等同于汇金买卖。
- 不把“汇金仓位区间”写成当前 ETF 持有人比例。
- 不把股指期货、中金所成交持仓排名、成交额、价格涨跌作为核心公式输入。
- 不把证金、社保、保险、其他机构统称纳入“中央汇金”口径。

## 2. 视频内容还原

### 2.1 核心逻辑

视频中的“助力汇金”可以概括为:

1. 选择中央汇金作为主要跟踪对象。
2. 选择 ETF 作为跟踪载体，因为 ETF 有公开份额数据和定期持有人披露。
3. 用基金定报中披露的汇金持有人数据建立基准。
4. 用交易所每日 ETF 总份额变化推导汇金持仓的可能区间。
5. 用份额连续、大幅变化作为强弱观察信号。
6. 用股指期货、多空变化、市场情绪、新闻等作为辅助验证。

视频强调一级市场 ETF 份额变化更关键，二级市场成交量、成交额和价格涨跌只能辅助。原因是 ETF 二级市场买卖不会改变 ETF 总份额，只有一级市场申购赎回会改变 ETF 总份额。

### 2.2 公式还原

变量定义:

| 变量 | 含义 | 口径 |
| --- | --- | --- |
| `S0` | 披露日 ETF 总份额 | 份 |
| `H0` | 披露日汇金合计持有份额 | 份 |
| `A` | 披露日汇金公开持仓比例 | `A = H0 / S0` |
| `X0` | 披露日非汇金公开持仓比例 | `X0 = 1 - A` |
| `S1` | 当前或目标交易日 ETF 总份额 | 份 |
| `B` | 当前总份额相对披露日总份额比例 | `B = S1 / S0` |
| `X` | 当前非汇金持有份额相对 `S0` 的归一化比例 | 无法直接观测 |
| `Y` | 当前汇金持有份额相对 `S0` 的归一化比例 | 估算区间 |

约束关系:

```text
A = H0 / S0
X0 = 1 - A
B = S1 / S0
B = X + Y
```

区间估算:

```text
Y_min = max(0, B - X0) = max(0, B - (1 - A))
Y_max = B
```

重要解释:

- `Y_min` 与 `Y_max` 是相对披露日总份额 `S0` 的归一化区间。
- 它不是汇金当前真实账户持仓。
- 它不是当前 ETF 持有人占比。
- 当 `S1 < H0` 时，不应直接判定为单位错误。如果披露日 `S0/H0/A` 已核验，则说明汇金旧披露持仓不可能原样保持，视频公式正是用这种变化推导区间。

### 2.3 操作纪律还原

买入观察条件:

- 市场出现较大回撤或风险释放机会。
- 汇金持仓 ETF 的总份额出现显著、连续增加。
- 视频提到的强信号是相对平常约 10 倍量、持续一周的份额增量。
- 股指期货、市场情绪、新闻和 ETF 份额变化需要交叉验证。

加仓观察条件:

- ETF 总份额继续扩大。
- 估算区间上移。
- 个人仓位上限参考估算区间上限，而不是价格涨跌本身。

卖出或减仓观察条件:

- ETF 总份额持续下降。
- 估算区间下降。
- 指数上涨但份额减少时，需要警惕二级市场价格上涨和一级市场赎回同时发生。

观望条件:

- 份额没有明显增加。
- 只有价格上涨但份额减少。
- 数据缺失、源日期异常或辅助信号不明确。

### 2.4 视频假设与限制

必须在产品中保留以下限制:

- ETF 份额变化不必然来自中央汇金，可能来自其他机构、套利交易、普通投资者申赎。
- “10 倍量持续一周”是视频作者经验阈值，不是官方规则。
- 股指期货多空变化的解释依赖套保假设，不能机械等同于现货买卖。
- 每次基金定报更新后，需要重新设定 `A`、`X0`、`S0`。
- V1 不自动输出“买入确认”信号；若后续实现“10 倍量持续一周”，必须先定义基准量口径，例如过去 N 个有效交易日的日份额变化中位数或均值，并在前端显示该阈值来源。

## 3. 数据源需求

### 3.1 核心数据源

| 数据 | 用途 | 优先级 | 要求 |
| --- | --- | --- | --- |
| 含持有人明细的基金定期报告 | 建立 `H0`、`S0`、`A` 基准 | P0 | 优先年报/半年报；季报仅在明确披露前十大持有人或汇金主体持有人信息时使用，且必须可追溯到原始公告 |
| 上交所 ETF 规模/份额 | 获取沪市 ETF 每日 `S1` | P0 | 保存源日期、源单位和归一化份额 |
| 深交所 ETF 规模/份额 | 获取深市 ETF 每日 `S1` | P0 | 不得用运行日期直接当交易日期 |
| 交易日历 | 判断有效 `S1` 日期 | P0 | 过滤周末、节假日和非交易日 |

### 3.2 辅助数据源

| 数据 | 用途 | 是否参与核心公式 |
| --- | --- | --- |
| 中金所股指期货成交持仓排名 | 辅助验证风险偏好和套保解释 | 否 |
| ETF 行情、价格、成交额、IOPV | 辅助观察折溢价与流动性 | 否 |
| 指数估值、融资、北向资金、行业资金流 | 辅助市场环境判断 | 否 |

### 3.3 当前数据源遗留问题

当前 `etffun` 的已知问题必须作为 P0 数据质量任务处理:

- `huijin_config.json` 只能作为迁移种子，不是最终可信基准。
- 深交所 ETF 列表接口可能没有明确源日期，必须结合交易日历或页面元数据校验。
- 周末运行采集时不得生成周末有效交易日份额。
- 深市汇金 ETF 样本存在历史份额缺口，需要优先回填和审计。
- `daily_snapshot` 缺少 source/audit 层，无法解释某条份额数据来自哪个接口、什么单位、何时抓取。

## 4. 数据结构需求

### 4.1 `huijin_baseline`

用途: 保存每只 ETF 的汇金披露基准。

必需字段:

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| `baseline_id` | VARCHAR | 基准唯一 ID |
| `code` | VARCHAR | ETF 代码 |
| `name` | VARCHAR | ETF 名称 |
| `report_period` | VARCHAR | 报告期，例如 `2025Y`、`2025H1` |
| `report_date` | DATE | 报告期截止日 |
| `disclosure_date` | DATE | 公告披露日期 |
| `s0_total_shares` | DOUBLE | 披露日总份额，单位为份 |
| `h0_total_shares` | DOUBLE | 汇金合计持有份额，单位为份 |
| `a_ratio` | DOUBLE | `H0 / S0` |
| `source_doc_title` | VARCHAR | 来源公告标题 |
| `source_doc_url` | TEXT | 来源公告 URL 或本地归档路径 |
| `source_doc_hash` | VARCHAR | 来源文档 hash，用于防止来源漂移 |
| `source_page` | VARCHAR | 页码或章节 |
| `verification_status` | VARCHAR | `draft`、`verified`、`rejected` |
| `verified_at` | TIMESTAMP | 校验时间 |
| `is_active` | BOOLEAN | 是否为当前默认基准 |

校验要求:

- `s0_total_shares > 0`
- `0 <= a_ratio <= 1`
- `abs(a_ratio - h0_total_shares / s0_total_shares)` 在容忍范围内
- `disclosure_date <= 计算日期`
- 未通过 `verified` 的记录不得用于前端结论展示

### 4.2 `huijin_baseline_holder`

用途: 保存逐主体持有人披露。

必需字段:

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| `baseline_id` | VARCHAR | 关联 `huijin_baseline` |
| `holder_name` | VARCHAR | 原文持有人名称 |
| `holder_group` | VARCHAR | 归一化主体组 |
| `holder_shares` | DOUBLE | 持有份额，单位为份 |
| `holder_ratio` | DOUBLE | 披露占比 |
| `source_line` | TEXT | 来源行摘要或位置 |

口径要求:

- 仅将“中央汇金投资有限责任公司”和“中央汇金资产管理有限责任公司”等明确汇金主体归入 `Central Huijin`。
- 不把证金、社保、保险、券商资管等其他机构混入中央汇金口径。

### 4.3 `daily_snapshot_audit`

用途: 保存份额采集的源数据和质量信息。

必需字段:

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| `date` | DATE | 归属交易日 |
| `code` | VARCHAR | ETF 代码 |
| `source_name` | VARCHAR | `sse_etf_scale`、`szse_etf_scale` 等 |
| `source_url` | TEXT | 请求 URL 或接口说明 |

| `source_date_inferred` | BOOLEAN | 是否由交易日历推断 |
| `raw_total_shares` | DOUBLE | 原始份额值 |
| `raw_unit` | VARCHAR | 原始单位，例如 `份`、`万份` |
| `normalized_total_shares` | DOUBLE | 标准化为“份”后的值 |
| `run_id` | VARCHAR | 关联采集运行 |
| `quality_flags` | VARCHAR | 质量标记集合 |

### 4.4 `market_calendar`

用途: 判断有效交易日、最近交易日、前一交易日。

必需字段:

| 字段 | 类型建议 | 说明 |

| `date` | DATE | 日期 |
| `is_trading_day` | BOOLEAN | 是否交易日 |
| `prev_trading_day` | DATE | 前一交易日 |

### 4.5 `data_source_run`

用途: 记录每次采集运行。

必需字段:

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| `run_id` | VARCHAR | 运行 ID |
| `task_name` | VARCHAR | 任务名 |
| `source_name` | VARCHAR | 数据源 |
| `started_at` | TIMESTAMP | 开始时间 |

### 4.6 `data_quality_issue`

用途: 记录可阻断或需提示的问题。

必需字段:

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| `issue_id` | VARCHAR | 问题 ID |
| `code` | VARCHAR | ETF 代码，可为空 |
| `date` | DATE | 相关日期，可为空 |
| `issue_type` | VARCHAR | 问题类型 |
| `severity` | VARCHAR | `blocker`、`warning`、`info` |
| `status` | VARCHAR | `open`、`resolved`、`ignored` |
| `message` | TEXT | 问题说明 |
| `created_at` | TIMESTAMP | 创建时间 |

建议问题类型:

- `NON_TRADING_DATE`
- `SOURCE_DATE_INFERRED`
- `MISSING_BASELINE`
- `MISSING_S0`
- `MISSING_S1`
- `SHARE_GAP`
- `ABNORMAL_JUMP`
- `UNIT_UNVERIFIED`
- `SOURCE_FAILED`

### 4.7 `fund_share_event`

用途: 记录份额折算、拆分、合并、代码变更、名称变更等影响 `S0` 与 `S1` 可比性的事件。

如果某 ETF 在 `report_date` 到 `as_of_date` 之间存在未处理的份额折算事件，汇金区间计算必须阻断或使用调整后的复权份额。

### 4.8 `huijin_watch_group`

用途: 支持同指数 ETF 池观察。

需求:

- 按沪深300、中证500、中证1000、创业板、科创50、上证50、上证180 等建立观察组。
- 一个观察组包含多只汇金持仓 ETF。
- 支持单 ETF 口径和同指数合并口径。
- 合并口径必须清楚标注，不得与单 ETF 持有人比例混用。

## 5. 功能需求

### 5.1 汇金观察概览

接口建议: `GET /api/huijin/overview`

每只 ETF 返回:

- 代码、名称、交易所、跟踪指数、观察组。
- 当前使用的基准报告期、报告日、披露日。
- `S0`、`H0`、`A`、最新有效 `S1`、`B`、`Y_min`、`Y_max`。
- 日、周、月份额变化。
- 数据质量状态和阻断原因。
- 来源文档标题、来源日期、校验状态。

展示规则:

- 只有 `verification_status='verified'` 且无 blocker 问题时展示区间。
- 如果基准缺失、源日期异常、交易日异常、单位未校验或份额断档，只展示“数据不足/需复核”。

### 5.2 单 ETF 详情

接口建议: `GET /api/huijin/{code}/series`

返回:

- 历史 `S1` 序列。
- `B/Y_min/Y_max` 历史序列。
- 份额变化强度。
- 质量标记。
- 披露基准变更记录。
- 份额折算事件。

### 5.3 同指数 ETF 池

需求:

- 支持按观察组聚合展示总份额变化。
- 聚合结果只能作为“ETF 池份额观察”，不能写成单一汇金账户仓位。
- 聚合口径需要单独展示组内 ETF 列表和每只 ETF 的质量状态。

### 5.4 前端展示

必须调整现有文案:

| 当前倾向 | 调整为 |
| --- | --- |
| 国家队增仓 | 汇金 ETF 份额观察 |
| 汇金动态调仓 | 汇金 ETF 份额观察 |
| 比汇金 | 份额变动强度 |

展示必须包含:

- 报告期。
- 披露日。
- 最新有效份额日。
- 是否使用推断源日期。
- 数据质量状态。
- “不是实时汇金持仓”的说明。

## 6. 计算规则

### 6.1 基准选择

计算某个 `as_of_date` 的区间时，只能使用:

```text
verification_status = 'verified'
disclosure_date <= as_of_date
is_active = true
```

如果回测历史日期，不能使用未来才披露的基准，避免未来函数。

### 6.2 `S1` 选择

`S1` 必须来自有效交易日:

- 优先使用 `as_of_date` 当天的有效份额。
- 如果当天无有效份额，可使用最近一个有效交易日，但必须标注 `stale`。
- 不得使用周末或节假日作为有效交易日。
- 不得复用 `/api/etf/all` 当前“价格优先行”的选择逻辑作为 `S1` 选择规则。

### 6.3 质量门禁

以下情况阻断区间计算:

- 缺 `S0`。
- 缺 `H0`。
- 缺 `A`。
- 缺有效 `S1`。
- 基准未验证。
- 源单位未验证。
- 存在未处理份额折算事件。
- 最新份额日期不是有效交易日。

以下情况允许展示但必须提示:

- `source_date` 由交易日历推断。
- 份额有历史缺口。
- 份额出现异常跳变。
- 辅助数据缺失。

## 7. 验收标准

### 7.1 数据完整性

- 每条用于计算的基准都有原始公告来源。
- 每条用于计算的份额都有 source/audit 记录。
- 所有份额统一为“份”，前端可换算显示“亿份”。
- 深市和沪市 ETF 均经过交易日校验。

### 7.2 公式正确性

必须通过单元测试:

```text
A = H0 / S0
X0 = 1 - A
B = S1 / S0
Y_min = max(0, B - (1 - A))
Y_max = B
```

### 7.3 前端边界

- 异常数据不展示结论型区间。
- 正常数据展示区间时必须写明“相对披露日总份额归一化口径”。
- 不使用“确认增仓”“确认减仓”“真实仓位”等文案。

### 7.4 实施同步

后续代码实现必须同步更新:

- `db/schema.py`

- `db/__init__.py`

## 8. 最终实施方案

### Phase 0: 数据审计

- 新增只读审计脚本或管理接口，检查基准完整性、交易日异常、周末落库、深市缺口、异常跳变和任务失败。
- 输出每只汇金 ETF 的 `data_quality_issue`。
- 不在审计完成前展示汇金区间。

### Phase 1: 基准数据结构

- 新增 `huijin_baseline` 和 `huijin_baseline_holder`。
- 将 `huijin_config.json` 迁移为 draft 种子。
- 人工逐只回到基金定报原文核验后，才标记为 `verified`。

### Phase 2: 份额源审计结构

- 新增 `daily_snapshot_audit`、`market_calendar`、`data_source_run`、`data_quality_issue`。
- 修正深交所源日期问题。
- 修正周末或非交易日落库问题。
- 优先补齐汇金 ETF 池历史份额缺口。

### Phase 3: 计算与 API

- 新增汇金计算查询。
- 新增 `/api/huijin/overview`。
- 新增 `/api/huijin/{code}/series`。
- 新增缓存和读写库同步测试。

### Phase 4: 前端展示

- 新增“汇金观察”视图。
- 重命名旧筛选预设和列名，避免过度推断。
- 在详情页展示来源、质量状态和公式解释。

### Phase 5: 辅助验证与回测

- 引入中金所股指期货成交持仓排名作为辅助面板。
- 支持同指数 ETF 池观察。
- 增加策略历史回测，但不得使用未来披露基准。

## 9. 视频一致性复核表

| 复核项 | 视频内容 | 文档处理 | 结论 |
| --- | --- | --- | --- |
| 公式 `A` | 汇金公开持仓占比 | 定义为 `A = H0 / S0` | 一致 |
| 公式 `X0` | 非汇金公开持仓占比 | 定义为 `X0 = 1 - A` | 一致 |
| 公式 `B` | 当前份额 / 披露日份额 | 定义为 `B = S1 / S0` | 一致 |
| 公式 `Y_min` | `B - (1 - A)` | 实现为 `max(0, B - (1 - A))` | 一致，并补充下限保护 |
| 公式 `Y_max` | 当前份额比例上限 | 定义为 `Y_max = B` | 一致 |
| 一级市场优先 | ETF 总份额变化是核心 | 核心公式只使用 ETF 总份额 | 一致 |
| 二级市场辅助 | 成交量、价格不能直接说明汇金仓位 | 行情数据只作为辅助 | 一致 |
| 股指期货辅助 | 中金所持仓排名用于辅助验证 | 不进入核心公式 | 一致 |
| 10 倍量持续一周 | 作者经验强信号 | 标为经验阈值，不作为官方规则 | 一致 |
| 份额变化归因 | 份额变化不必然来自汇金 | 文档明确不能等同汇金买卖 | 一致 |
| 汇金仓位口径 | 视频沿用“仓位”说法但实际为估算 | 文档明确为相对 `S0` 的归一化区间 | 一致，并降低误读风险 |
| 定报更新 | 需要重新设定 `A/X0/S0` | 基准按报告期和披露日版本管理 | 一致 |
| 操作纪律 | 买入、加仓、减仓、观望依赖份额和辅助验证 | 功能只提供观察和质量门禁 | 一致 |

### 复核结论

本需求文档与视频内容保持一致，同时对视频中容易误读的部分增加了产品边界:

- “汇金仓位”在系统中只作为估算区间，不作为真实账户持仓。
- ETF 份额变化只作为公开数据观察，不作为汇金真实买卖确认。
- 股指期货和成交量只作为辅助验证，不参与核心公式。
- 当前项目已有汇金字段只能作为迁移种子，不能直接作为可信基准上线。
